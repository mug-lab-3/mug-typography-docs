APIリファレンス

Mug TypographyのLuaスクリプトから利用できるコールバック、ctxオブジェクト、座標系、mt.*ユーティリティの完全な仕様です。

対象: Mug Typography のスクリプティング機能でアニメーションを作るユーザー API level: 2

Mug Typography のスクリプティングは、プラグインの Script Text に書いた Lua 5.4 スクリプトで テキストレイアウトと文字・パーツのアニメーションを制御する機能です。開発には VSCode 拡張 「Mug Typography Scripting Simulator」(MT Sim) を使うと、ホストアプリを起動せずにプレビュー・ デバッグできます。

1. 基本モデル ​

スクリプトには最大3つのコールバック関数を定義します。どれも省略可能で、定義しなかった コールバックは単にスキップされます。

text
スクリプトのロード
  ↓
OnInitialize(ctx)     インスタンス定数の構築(mt.storageへ)
  ↓ (毎フレーム)
OnPreLayout(ctx)    レイアウト・全体・カメラの基礎値(Base)を変更
  ↓
文字・パーツのレイアウト生成
  ↓
OnLayout(ctx)         生成された文字・パーツを個別に調整
  ↓
描画

重要なのは「Baseを確定し、そのBaseから生成されたLayoutを調整する」という二段階です。

  • 文字数やパーツ数に影響する値(テキスト、フォント、字送りなど)は OnPreLayout で変更します
  • 個々の文字・パーツの位置・回転・色などは OnLayout で変更します
  • OnLayout から Base(ctx.global / ctx.camera)の変更（書き込み）はできませんが、OnPreLayout 通過後の確定値が読み取り専用（Read-Only）として参照できます

end

変更はフレーム内でのみ有効 ​

ctx は毎フレーム、ホストのインスペクタ値から作り直された状態でコールバックに渡されます。 コールバックで書き込んだ値はそのフレームの描画にだけ使われ、次のフレームではまたインスペクタ値 から再スタートします。インスペクタやキーフレームへ書き戻されることもありません。

たとえば ctx.global.rotation = ctx.global.rotation + 1.0 は「毎フレーム1度ずつ回り続ける」 のではなく、「常にインスペクタ値+1度」になります。回転を進めたい場合は ctx.global.rotation = ctx.time * 20.0 のように ctx.time から現在の角度を直接計算します。

2. フレーム独立性の要件（ステートレス実行） ​

ホストはフレームを時系列順に評価するとは限りません。スクラブ、逆再生、キャッシュ、並列レンダーの どの場合でも同じ結果になる必要があります。各フレームの結果は現在の入力値だけから決まり、 前フレームの結果やフレームの評価順序に依存してはいけません。本書では、この性質を フレーム独立性と呼びます。各コールバックは、この要件を満たす純関数として書きます。

  • 前フレームの結果をグローバル変数に保存しない
  • コールバック内でグローバル変数を書き換えない
  • トップレベルでは定数と補助関数を定義してよいが、フレーム評価中に変更しない
  • ctx や ctx.chars[index] などの参照をコールバック終了後まで保持しない
  • 文字数、パーツ数、特定フォントの形状を固定値として仮定しない
  • 逐次シミュレーションではなく、ctx.time から直接計算できる閉形式で書く
  • 文字・パーツの走査は必ず ctx.char_count / ctx.part_count を使う

文字・パーツごとに安定した乱数が必要なときは math.random ではなく mt.random(seed, index) を使います。math.random は整数フレーム・コールバックごとに 再シードされるため、呼び出し順に依存し、サブフレームで滑らかに変化せず、文字数や呼び出し回数の 変更によって後続要素の値も変わります。

インデックス ​

  • ctx.chars と ctx.parts は Lua 標準どおり 1始まり です
  • ctx.parts[index] の index は文字ごとの番号ではなく、フレーム全体を通したグローバルなパーツ番号です
  • フォントやテキストが変わると、文字数・パーツ数・対応する index も変わります

スクリプトメタデータ ​

スクリプトファイルの先頭に -- @<name> <value> の形式でコメントを記述することで、MT Simの 動作や初期設定を指定できます。これらはVS Code拡張／MT Sim用であり、ホストプラグインが コールバック実行時に解釈するLua APIではありません。

  • スクリプト識別:
    • -- @MugTypography（または -- @mt / -- @mug）: 先頭から5行以内に置くと、MT Sim がこのファイルを Mug Typography 用スクリプトとして認識します（入力補完・シミュレーター連携の前提）。配布サンプルはすべてこの行を含みます。
  • 再生時間指定:
    • -- @duration <秒数>: シミュレーターの再生時間の初期値を設定します（例: -- @duration 5.0 で5秒に設定）。
  • 必須機能の有効化 (-- @required <機能>):
    • -- @required 3d または 3dprojection: シミュレーター起動時に 3D 投影設定 (Enable 3D projection) を自動的に有効化します。
  • 推奨初期設定 (-- @recommend <設定>):
    • -- @recommend bg <指定>: 背景の初期値を設定します。#RRGGBB（または #RRGGBBAA）で単色、checkerboard（または checker）でチェッカー背景です。同一行に両方を書けます（例: -- @recommend bg #1e1e1e、-- @recommend bg checkerboard、-- @recommend bg checkerboard #ffffff）。スクリプトが塗り・ストローク・グラデーションなどで色を決めている場合、その色が識別しやすい高コントラストな背景を選ぶのが目的です（明るい文字には暗い背景、暗い文字や半透明表現にはチェッカーなど）。色を使っていないスクリプトでは指定不要です。
    • -- @recommend pos <X> <Y>: グローバル文字位置 (positionX, positionY) の初期値を設定します（例: 中央は -- @recommend pos 0.5 0.5、画面下部の字幕は -- @recommend pos 0.5 0.18）。Canvas正規化座標と同様に、Xは大きいほど右、Yは大きいほど上です。そのため、下部へ配置する場合は 0.5 より小さいYを指定します。スクリプトが h_align / v_align などでアラインメントを決めている場合、既定の中央寄せのままでは意図どおり画面内に収まらないことがあるため、画面内に収まる位置を合わせて指定するのが目的です。アラインメントを触っていないスクリプトでは指定不要です。
    • -- @recommend lang <言語コード>: スクリプト内で日本語などの特定言語のテキストを明示的に指定・想定している場合のみ指定します（例: -- @recommend lang ja。指定された場合、システムフォントから対応言語のフォントを自動選択します。テキストを固定指定しない汎用スクリプトでは指定不要です）。
    • シミュレーターインスペクタの Reset ボタンを押した際も、スクリプトのメタデータ指定に従って各値がリセットされます。

@duration / @required / @recommend は先頭から10行以内だけが解析対象です。複数の @recommend行にも、1行へ複数設定を書いた形式にも対応します。生成コードでは識別コメントを 1行目、その他のメタデータをその直後に置いてください。

AIコード生成の必須規約 ​

仕様書だけを入力としてコードを生成する場合は、次を既定方針とします。

  • Baseを変える処理はOnPreLayout、文字・パーツ・出力を変える処理はOnLayoutへ書く
  • 直接添字アクセスする前にctx.char_count / ctx.part_countで存在を確認する
  • アニメーション結果は毎フレームctx.timeまたはctx.timelineから直接計算する
  • Inspectorのraw値を維持する場合は、固定値で上書きせず現在値へ加算・乗算する
  • 個別塗り色を採用するときはfill.use = trueも設定する
  • 固定速度のintro/outroには実秒とmt.timeline.intro_outro_seconds / remainingを使う
  • 3D固有処理はif ctx.meta.capabilities.has_3d then ... endで囲む
  • 個別scale / stretchを変更して再組版する場合は、そのフレームの変更後にreflowを呼ぶ
  • offset_*と個別pivot_*は絶対位置ではなく、0.5を無変位とする変位量として扱う
  • 要素別乱数はmath.randomではなくmt.random / mt.noise*を使う
  • I/O、モジュール読込、OS API、コルーチン、デバッグAPI、コールバックからのグローバル書き込みを使わない
  • 公開numberフィールドへはNaNや±infinityではなく有限値だけを書く
lua
-- @MugTypography

function OnPreLayout(ctx)
    ctx.global.rotation = ctx.global.rotation + ctx.time * 20.0
end

function OnLayout(ctx)
    for index = 1, ctx.char_count do
        local character = ctx.chars[index]
        local progress = mt.stagger(ctx.time, index, 0.08, 0.4)
        character.offset_y =
            character.offset_y + (1.0 - mt.ease.out_cubic(progress)) * 0.2
    end
end

3.1 OnInitialize(ctx) ​

インスタンス固有の不変値を mt.storage に構築する。「最初のフレーム」ではなく、 インスタンスまたはスクリプトソースの初期化ごとに1回呼ばれる。テキストやフォントの変更では 再実行されない。

ctx.text / ctx.char_count / ctx.part_count は 未定義（フォント名も ctx.global.font_name としてはまだ読めない。 ctx.font_name というフィールドは存在しない）。 テキスト・フォントは毎フレームの Base（OnPreLayout の ctx.text / ctx.global.text / ctx.global.font_name）で扱い、shaped な文字数・パーツ数は OnLayout でのみ存在する。 文字ごとの決定論的ばらつきは OnLayout で mt.random(ctx.meta.instance_seed, index) を使う（§4.3）。

lua
function OnInitialize(ctx)
    mt.storage.amplitude = 0.06
end

完了後の mt.storage は再帰的に凍結され、以後のコールバックからは読み取り専用になる。 格納できる値は boolean / number / string とその入れ子 table（キーは number または string）で、 循環参照と64階層以上の入れ子は初期化エラーになる。

3.2 OnPreLayout(ctx) ​

ホストパラメータから作られた Base の作業コピーを受け取り、レイアウト生成前の値を変更します。 テキストを書き換えるときは読み取り専用の ctx.text ではなく ctx.global.text へ代入します。

TIP

スケール調整の設計ルール: テキスト長や文字数に応じた全体のサイズ調整（画面外へのはみ出し防止など）は、OnLayout 内で個々の文字の character.scale を動的に縮小するのではなく、OnPreLayout フェーズで ctx.global.scale をあらかじめ調整してください。これにより、文字が徐々に追加されるアニメーション（タイプライターなど）の実行中に、全体のスケールが文字数の増減に合わせてリアルタイムに変動してガタつくのを防ぐことができます。

時間軸（実秒数 vs 進行度）の設計ルールは §4.1 を参照してください。

3.3 OnLayout(ctx) ​

確定した Base から生成された文字・パーツを調整します。文字数とパーツ数はこの時点で確定しています。

時間軸の設計ルール（アンカー × 速度） ​

アニメーションの時間設計は「①どこに固定するか（アンカー）」と「②どの速さで進めるか（速度）」の 2軸で決めます。**速度の既定は実秒（ctx.time ベース）**で、クリップ長に比例させる progress を 使ってよいのは後述の「全体スパン型」だけです。

パターンアンカー書き方
イントロ（登場）クリップ先頭ctx.time をそのまま使う（実秒）。イントロ＋アウトロ一括なら mt.timeline.intro_outro_seconds(§10.9)
アウトロ（退場）クリップ終端mt.timeline.remaining(ctx)（終端までの残り実秒。§10.9）
タイプライター・stagger・ばね等クリップ先頭ctx.time（実秒。1文字あたり◯秒など物理速度で指定）
全体スパン型（尺いっぱいのドリフト・スイープ・リビール）クリップ全体progress（尺が伸びたら動きも伸びるのが期待動作）

トランジションを progress の割合で書くと、尺を伸ばしたときにフェードやタイピングが スローモーション化して破綻します。アウトロも本質は「終端に揃えること」であって速度を尺に 比例させることではないため、終端アンカーの実秒で書きます。

lua
-- NG: typing speed changes when the clip length changes
local visible = math.floor(mt.timeline.progress(ctx) * ctx.char_count)

-- OK: constant 0.08s per character regardless of clip length
local visible = math.floor(ctx.time / 0.08)

-- OK: outro fade over the last 0.5s, anchored to the clip end
local fade = mt.clamp(mt.timeline.remaining(ctx) / 0.5, 0.0, 1.0)

-- OK: 0.8s intro + 0.5s outro at fixed real-time speed
-- (compressed automatically only when the clip is shorter)
local intro, outro = mt.timeline.intro_outro_seconds(ctx, 0.8, 0.5)

3Dプリセットではこの積は近似にしかなりません。 カメラ投影が奥行き(z、 yaw/pitchによる見かけの奥行き変化、カメラの perspective / zoom を含む)に応じた 遠近除算を行うため、画面上サイズは奥行きに対して非線形に変化し、傾いた文字は 頂点ごとに歪みが生じて単一の倍率では表せません。3Dプリセットで正確な値が必要な 場合は、上記の ctx.font(または character.geometry、§6.2)から実測値を読み取って ください。

WARNING

シャドウを有効にすると、本体とは別にシャドウ形状を描画するため処理負荷が増えます。 特に、描画対象で chars[index].shadow.use または parts[index].shadow.use が1つでも true になると、個別色用のCPUシャドウ経路になります。影色のアルファと対象の合成 opacityがすべて1なら直接描画する1パス経路になります。それ以外は完成アルファ マスクと最前面の色所有者マップを生成します。この経路の描画回数は個別色の種類や 個数では増えませんが、通常のグローバルシャドウより重くなります。

負荷は主に、出力解像度、描画中のパーツ数とパス形状の複雑さ、太いストローク、 広く離して配置・変形した文字、バウンディングボックス、および3D投影などによって シャドウ処理領域が大きくなるほど上がります。大きな distance も構図や描画経路に よっては処理領域やGPUオーバーレイのパディングを広げます。

shadow.use = true は、個別色が継承元と同じ場合や、対象の opacity が0の場合でも CPU個別色経路を有効にします。解決後の影色がすべて同じでも通常のグローバル シャドウ経路には縮約されません。不要なCPU処理やGPUシャドウ経路からの切り替えを 避けるため、実際に異なる影色が必要な文字・パーツ、または ctx.global.shadow.enabled = false のまま特定の対象だけに影を付ける場合にだけ shadow.use を有効にしてください。すべて同じ影色でよい場合は、個別のshadow.useを falseにして ctx.global.shadow.enabled と ctx.global.shadow.color を使う方が軽量です。

グラデーションが塗るのは文字・パーツの fill.use がどちらも false の部分だけである。 fill.use が true の個別色はグラデーションより優先され単色になる。 fill.use と opacity の段階分けは §8。

7.2 座標 ​

  • 位置・offset・pivot の単位定義は §7.0
  • X は canvas 幅、Y は canvas 高さを基準にします
  • Y は上方向が正です(Lua値のYを増やすと画面上方向へ移動)

WARNING

offset の変位量は文字サイズではなく canvas 全体を基準にしています。offset_x などに加算する数値を「文字1個分に対して十分小さいだろう」という感覚だけで 決め打ちすると、実際には canvas 幅基準で大きすぎる値になり、隣接する文字・パーツと 視覚的に重なることがあります。0.02 は常に canvas 幅の2%であり、フォントサイズや canvas解像度に関わらず一定の比率です。文字送り幅(geometry.advance_x)がcanvas幅の 2%を下回るフォント・字送り設定では、この値だけで簡単に隣接文字へ食い込みます。 ジッターやノイズで小さな変位を加える場合は、絶対値を決め打ちせず character.geometry.advance_x(文字送り幅の正規化値、§6.2)と比較し、その何分の1かに 収まる値を使ってください。

WARNING

回転(rotation)も文字サイズとの相対関係で「大きすぎる」ことがあります。 文字の字面サイズに対して文字送り幅の余白が小さいフォント・字送り設定では、 各文字を独立に数度回転させるだけでも、回転した角が隣接文字の領域へ食い込んで 重なって見えることがあります。これは rotation 自体の不具合ではなく、字間の 余白に対して回転の振れ幅が大きすぎるという幾何学的な制約です。振れ幅を大きく 取りたい場合は ctx.global.tracking で字間を広げるか、振れ幅そのものを 小さくしてください。

  • 3Dの奥行き z は奥方向が正です（値が大きくなるほどカメラから遠ざかり、画面奥に配置されます。負の値で手前に近づきます）
  • 文字・パーツの offset / pivot は canvas 正規化変位である（§7.0）。geometry の canvas 座標および global.position / global.pivot は canvas 正規化位置である
  • geometry の canvas 位置は canvas 左下が 0,0、中心が 0.5,0.5
  • geometry の local 位置と advance は変位量なので中心オフセットを持たず、Y-up で表されます
  • rotation、yaw、pitch、angle はすべて degree

WARNING

角度系の符号は「スクリーン感覚（時計回りが正、90度 = 真下）」で統一されており、 座標系（Y-up）とは向きの感覚が逆です。 offset_y は上が正なのに対し、rotation / gradient.angle / shadow.angle は正で時計回りに回り、シャドウ角 90 は真下を 指します。Y-up の数学的慣習（反時計回りが正、270 = 真下）を持ち込むと、影が真上に 出る・回転が逆に回る、という取り違えになります。

TIP

円や斜めの空間場を geometry の canvas 座標から作るときは、正方形ピクセル (ctx.canvas.pixel_aspect_ratio == 1)ならXにctx.canvas.aspect_ratioを掛けて 正方形化した座標系で距離・内積を取ってください。bounds_center_x は canvas 幅比、bounds_center_y は canvas 高さ比のため、横長 canvas では そのまま sqrt(dx²+dy²) を取ると円が横に潰れます。例: local worldX = (geometry.bounds_center_x - 0.5) * ctx.canvas.aspect_ratio、 local worldY = geometry.bounds_center_y - 0.5。 非正方形ピクセルの物理表示比で計算する場合は、X係数を ctx.canvas.aspect_ratio * ctx.canvas.pixel_aspect_ratioとします。 ショーケースの 34_light_sweep_prism.lua や 51_ripple_echo.lua がこのパターンです。

  • 2D回転（rotation）: 正（+）が時計回り（右回り）、負（-）が**反時計回り（左回り）**です。
  • 3Dオブジェクト回転（yaw, pitch）: Individual/Global ともに、個別ピボットを中心とした回転です。
    • yaw: 正（+）でオブジェクトの右端が手前（-Z）へ、左端が奥（+Z）へ回転します（上から見て時計回り）。
    • pitch: 正（+）でオブジェクトの上端が手前（-Z）へ、下端が奥（+Z）へ回転します（手前に倒れ込むように回転する、右から見て時計回り）。
  • カメラの旋回（yaw, pitch）: 注視点の周りをカメラが公転（Orbit）します。
    • yaw: 正（+）でカメラが注視点の周りを右に回り込みます（画面上のオブジェクトは左に回転して見えるようになります）。
    • pitch: 正（+）でカメラが注視点の周りを上に回り込みます（見下ろすアングルになります）。
  • グラデーション角度（angle）: 0度で左から右（Xの正方向）に進み、角度が増える（正）と時計回りに回転します。
  • シャドウ角度（angle）: 0度で右方向（Xの正方向）に伸び、角度が増える（正）と時計回りに回転します（90度で真下方向）。

pivot は「絶対位置」と「変位」の2種類がある ​

  • ctx.global.pivot_x/y は canvas 上の絶対位置です(0.5, 0.5 = canvas中央)
  • 文字・パーツの pivot_x/y は自身の基準点からの変位です。0.5, 0.5 のとき、文字は 自然なバウンディングボックス中心(geometry.bounds_center 相当)、パーツは自然位置 (geometry.canvas_center 相当)を中心に回転・スケールします
  • したがって文字・パーツの pivot に geometry の canvas 座標をそのまま代入してはいけません。 「自身の中心から右へ10%ずらした点を中心に回す」なら pivot_x = 0.5 + 0.1 です
  • 文字の辺を支点にするときは、自然中心からの半サイズ分だけ変位させます （bounds_width / bounds_height は canvas 軸比で、pivot の変位と同じ単位系です）:
    • 下辺: pivot_y = 0.5 - geometry.bounds_height * 0.5
    • 上辺: pivot_y = 0.5 + geometry.bounds_height * 0.5
    • 左辺: pivot_x = 0.5 - geometry.bounds_width * 0.5
    • 右辺: pivot_x = 0.5 + geometry.bounds_width * 0.5

8. 継承の仕組み ​

use / opacity / fill.transparent の役割 ​

フィールド役割
fill.use / stroke.use / shadow.use継承解決で raw 個別値を採用するか（true）上位を継承するか（false）。表示/非表示の制御ではない
opacity（global / character / part）解決済みの色（グラデーション含む）に対するアルファ乗算。fill.use の判定には関与しない
global.fill.transparent塗りを透明化する。ストローク等は残る

描画パイプライン上の段階:

  1. 色の決定: fill.use が true なら raw 個別色、false ならグラデーション（有効時）または Global 塗り
  2. アルファ乗算: 1 で決まった色に global.opacity × character.opacity × part.opacity を乗算

use = false のとき同じグループの raw 色を書き換えても描画には採用されない。 opacity のみの変更はグラデーション選択を維持したまま不透明度だけを変える。

描画に使われる値は次の順で決まる。

text
OnPreLayout で確定した Global
  ↓
文字: 各グループの use が true なら raw 個別値、false なら Global 値
  ↓
パーツ: 各グループの use が true なら raw パーツ値、false なら文字の解決済み値

gradient.color_space が none 以外のときは、このチェーンで「Global 値まで落ちた」 (文字・パーツどちらの fill.use も false の)部分だけがグラデーションで塗られ、 個別色が有効な文字・パーツは raw 個別値の単色で塗られます。

パーツの stroke / shadow はホスト側にパラメータグループが存在しない(常に未使用/既定値から 始まる)点を除けば、fill と全く同じ継承チェーンに従います。

OnLayout に渡される文字・パーツは継承前のraw値です。Lua からは各グループの use も 変更できるため、個別値を書き換えて同じコールバック内で有効化できます。

lua
local character = ctx.chars[index]
character.fill.use = true
character.fill.color = mt.color.from_hsv(index / ctx.char_count, 0.8, 1.0)

transform系フィールド(offset / scale / rotation など)には use_* がなく、常に描画へ反映されます。 スクリプトによる変更はそのフレーム内だけで完結し、ホストのInspectorやキーフレームへ書き戻される ことはありません。

mt.spring(t, frequency, damping) ​

減衰振動を閉形式で評価します。t = 0 で 1 から始まり、振動しながら 0 へ収束します。 「行き過ぎて揺れながら収まる」動きを、フレーム間の状態なしで作れます。

引数説明

戻り値: 減衰振動値（初期値1、収束値0）。実装は exp(-damping * t) * cos(2π * frequency * t)（t > 0 のとき）です。

WARNING

戻り値は 0–1 に収まりません。 減衰中に 負の値（および、位相によっては 1 をわずかに 超えない範囲での行き過ぎ）を取り得ます。これはオーバーシュート表現そのものであり、 mt.ease.out_back / elastic が 0–1 の外へはみ出すのと同種です。

  • 向いている用途: offset / rotation / 散らばり量などの変位の残差 （例: 0.5 + settle * 0.1）
  • 向いていない用途: 進行度・opacity・scale / stretch へのそのままの代入や、 mt.lerp の t としての利用。進行度が必要なら mt.ease.* や mt.saturate を使い、 負値をそのまま倍率に掛けないでください（stretch はホスト側の下限クランプがなく、 負のまま描画に渡り得ます。scale は正の下限へクランプされ、opacity は描画直前に 0–1 へクランプされます）

mt.ease.<name>(t) は進行度 t(通常0–1)を受け取り、緩急のついた進行度を返します。 mt.stagger や mt.cycle が返す直線的な進行度に表情を付けるのが典型的な使い方です。

  • in_out_ — 前半加速・後半減速

Quadin_quad / out_quad / in_out_quad最も緩やかな加減速

引数説明

API Level 2の再組版関数です。横書きではgeometry.canvas_origin_*、縦書きでは geometry.vertical_origin_*を使い、隣接する自然原点の差からシェーピング・kerning・tracking・ margins由来の間隔を保持したまま、現在のscale * stretchを送りへ反映します。個別rotationと pivotも原点整列へ含まれます。可視インク輪郭間の距離を均一化する関数ではありません。

function OnPreLayout(ctx)

付録A: 個別 scale 後の再組版 ​

レイアウトは全文字等倍前提で送りを決めたあと、各文字を geometry.bounds_center（pivot 0.5） 周りに scale する。そのため文字ごとに scale だけ変えるとベースラインがずれ、送りが崩れる。 API Level 2の再組版はmt.layout.reflow（§10.6）が行う。

pivot 0.5 のとき、自然座標上の点 P の写像（canvas 正規化・Y-up、C = bounds_center、 s = scale×stretch、d = offset−0.5）は次式である。

text
P' = C + s × (P − C) + d
d  = P'(目標) − C − s × (P − C)
  • 交差軸（ベースライン）: P = geometry.canvas_origin
  • 送り軸: セル左端も canvas_origin。セル幅は s × geometry.advance_x（tracking/margins 不含）

上式はscaleだけを説明する簡略形です。Level 2では横組み原点と縦組み原点を両方公開し、 mt.layout.reflowがrotation・pivot・非正方形canvasを含む完全な2D文字変換で組版原点を 整列します。Deprecatedのmt.layout.retypesetだけが回転0・pivot 0.5相当の旧簡易計算です。

付録B: グラデーションの適用領域と座標計算 (gradient coordinate system) ​

グラデーションの start_position と end_position がどのように解釈され、どの領域を基準に座標計算されるかの詳細です。

B.1 基準となるバウンディングボックス ​

グラデーション全体の描画範囲は、OnLayout で文字やパーツが個別に変形（scale, rotation, offset など）された**後のテキスト全体のバウンディングボックス（BB）**を基準とします。

  • 2D描画時: 変形後の全文字・パーツのパスを囲む canvas 上の2Dバウンディングボックス。
  • 3D投影時: カメラ投影の影響を受けない 3D レイアウト平面（plane）上におけるバウンディングボックス（キャラクターごとに個別に投影座標が計算され、カメラが動いてもグラデーションの位置が文字に対してズレないように制御されます）。

B.2 角度に沿った投影による基準線（0.0 〜 1.0）の決定 ​

グラデーションの基準線（0.0 〜 1.0 の区間）は、単なるバウンディングボックスの矩形サイズ（幅や高さ）ではなく、グラデーションの angle（角度）方向にバウンディングボックスを正射影したときの最近端と最遠端を結ぶ線分として決定されます。

具体的には、以下のアルゴリズムで基準線 [bbP0, bbP1] を求めます：

  1. バウンディングボックスの中心座標 $C(cx, cy)$ を求めます。
  2. グラデーション角度 angle を方向ベクトル $D(dx, dy) = (\cos(\theta), \sin(\theta))$ に変換します（$\theta$ はラジアン換算した角度）。
  3. バウンディングボックスの4つの頂点 $P_i \ (i = 1..4)$ について、中心 $C$ からの相対ベクトル $V_i = P_i - C$ を求め、方向ベクトル $D$ との内積（射影の長さ）を計算します。 $$\text{proj}_i = V_i \cdot D$$
  4. 4つの射影値のうち、最小値を $\text{minProj}$、最大値を $\text{maxProj}$ とします。
  5. 基準線の開始点 bbP0 と終了点 bbP1 を以下のように決定します： $$\text{bbP0} = C + D \cdot \text{minProj}$$ $$\text{bbP1} = C + D \cdot \text{maxProj}$$

これにより、例えば angle = 0 (右方向) の場合、bbP0 はバウンディングボックスの左端、bbP1 は右端になります。

NOTE

角度の向きと色の割り当てangle は 7.2 座標 の定義どおり、画面基準で 0度 = 右方向・正（+）で時計回りです（90度 = 真下方向）。基準線は bbP0 が angle の逆方向側の端、bbP1 が angle の方向側の端になり、色は start_position 側（bbP0 寄り）に fill 色、end_position 側（bbP1 寄り）に end_color が割り当てられます。

  • angle = 90（真下方向）の場合: bbP0 が上端、bbP1 が下端になり、fill 色が上端・end_color が下端のグラデーションになります。
  • angle = -90 または 270（真上方向）の場合: bbP0 が下端、bbP1 が上端になり、fill 色が下端・end_color が上端のグラデーションになります。

斜め方向の場合も同様に、指定角度の方向に最も突き出た頂点と最も引っ込んだ頂点の位置が自動的に 0.0 と 1.0 の境界として機能します。

B.3 start_position / end_position による補間 ​

実際のグラデーションの塗りの開始点 P0 と終了点 P1 は、上記で決定された基準線 [bbP0, bbP1] を start_position ($t_s$) および end_position ($t_e$) の比率で線形補間（または外挿）した座標になります。

$$\text{P0} = \text{bbP0} + (\text{bbP1} - \text{bbP0}) \cdot t_s$$ $$\text{P1} = \text{bbP0} + (\text{bbP1} - \text{bbP0}) \cdot t_e$$

  • start_position = 0.0 のとき、グラデーションの始点は角度 angle に沿った最も手前（逆方向）のテキスト端に一致します。
  • end_position = 1.0 のとき、グラデーションの終点は角度 angle に沿った最も奥（同方向）のテキスト端に一致します。
  • デフォルト値は通常 start_position = 0.2、end_position = 0.8 となっており、テキスト端の内側でなだらかに変色するようになっています。
